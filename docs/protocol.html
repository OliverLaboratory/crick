Protocol specification

How the crick proof-of-work blockchain works, end to end.

Implements DIPS v2 (arXiv:1911.00435) + the multi-puzzle extension (arXiv:1708.09419 §III). Matches reference implementation v0.1.0.

1Overview

crick is a Bitcoin-style proof-of-work blockchain with one change to the mining rule: a block can be produced either by ordinary hashing, or by including an improved solution to a scientific problem, in which case it is accepted at a reduced hashing difficulty. Solving is therefore the rational way to earn a block, and the same effort that secures the chain also produces useful results.

The design has three pillars, each detailed below:

  • Dual difficulty. Two difficulties coexist: d_b for ordinary (“classical”) blocks and d_r < d_b for blocks carrying a better solution.
  • Independent retargeting (DIPS v2). Each difficulty is retargeted from the production rate of its own block type, and a “drought” of unproductive blocks deepens the discount.
  • Epochs. The chain works one problem instance at a time, drawn from a large corpus, rotating to the next when the current one is exhausted.

Everything on the consensus path is integer arithmetic and a deterministic function of the block sequence, so any node replaying the chain reaches bit-identical state.

2Blocks & hashing

A block has the following fields:

FieldMeaning
heightIndex in the chain; genesis is 0.
prev_hashHash of the parent block (links the chain).
timestampUnix seconds the block was produced.
minerAddress that receives the block reward.
difficultyThe difficulty this block claims to satisfy (d_b or d_r).
transactionsOrdered list; the first must be the coinbase.
solutionA puzzle solution (any JSON value), or null for a classical block.
problemGenesis only: the corpus spec {type, seed, corpus_size}. Must be null in every other block.
nonceThe value miners vary to satisfy the target.

Block hash

Hashing uses double SHA-256 (sha256d) over a canonical JSON serialization of every field except the nonce, with the nonce appended. Canonical JSON sorts keys and uses compact separators, so the byte string is identical on every node.

base = canonicalJSON({height, prev_hash, timestamp, miner, difficulty, transactions, solution, problem})
hash = sha256d( base + ":" + nonce )

Separating the nonce lets a miner grind nonces without re-serializing the block body.

3Mining: two paths

To extend the tip, a miner produces a block one of two ways:

  1. Classical. solution = null. The block must satisfy the base difficulty d_b (standard Bitcoin-style hashing).
  2. Solution. solution is a strictly better answer to the active problem instance than anything previously recorded for it. The block then only needs to satisfy the reduced difficulty d_r.

Because d_r < d_b, finding an improvement is cheaper than out-hashing the network at full difficulty — so miners are paid to do science. A miner runs a solver and a hasher together: if the solver finds a better solution it mines at d_r; otherwise it mines classically at d_b.

4Proof of work & the target

Difficulty d maps to a 256-bit target; a block is valid when its hash, read as a big-endian integer, is below that target. Higher difficulty ⇒ smaller target ⇒ more hashing.

target(d) = floor( (2²⁵⁶ − 1) / max(d, 1) )
valid ⇔ int(hash, 16) < target(d)

A block declares which difficulty it targets in its difficulty field; validators independently recompute the required value (see §14) and reject any mismatch, so a miner cannot claim d_r without a qualifying solution.

5Problems: the verifier contract

A problem is any object implementing four operations. The contract is what makes a problem safe to use as proof of work:

OperationContract
verify(s)Cheap, exact, deterministic check that s is a well-formed, valid solution. This is the only thing every node runs, so it must be fast.
score(s)An integer where higher is strictly better. score(null) is a floor below any real solution. No floating point — so “is this an improvement?” is an exact comparison every node agrees on.
improve(best)The (expensive) miner-side search for something better than best. Never trusted; only its output is verified.
generate(seed)Deterministically construct instance from a seed string (used to build the corpus).

The essential asymmetry: verifying is cheap, finding is hard. A solution block forces the network to do one verification, but the miner had to search to find it. A strictly-better requirement also neutralizes copying — re-submitting a known solution is never valid, since it does not beat the recorded best.

6The built-in problems

Choose one at genesis with crick init --problem <type>.

docking — protein–ligand docking (default)

Find a ligand pose (a rigid placement) with lower energy than the best so far, scored by a fixed energy function on an integer 3D lattice. A pose is {t:[x,y,z], rot:k}: an integer translation plus one of the 24 integer rotation matrices of the cube. Because rotations are integer matrices, placed atom coordinates stay exact integers.

E = Σligand a Σprotein p pot( ‖a − p‖² )   (integer)
pot(d²) = CLASH_PENALTY if d² < 4;  contact(typeₐ, typeₚ) if d² ≤ 16;  0 otherwise
score = −E  (lower energy ⇒ higher score)

Verifying a pose is one O(|ligand|·|protein|) integer sum; finding a low-energy pose means searching rotations × translations. Poses must keep the ligand inside the binding box.

mcs-protein — maximum common subgraph

The largest substructure shared by two protein contact graphs — which is exactly the maximum clique of their modular product graph. A solution is a list of residue↔residue pairs; score is the number of matched residues. Verifying checks the mapping is a clique in the product graph (O(k²) edge lookups); finding it is NP-hard.

max-clique — reference problem

Largest clique in a pseudorandom G(n, p) graph derived from the instance seed. Trivial exact verification and a short improvement ladder; used to exercise the protocol itself.

7Difficulty retargeting (DIPS v2)

crick uses DIPS protocol v2, “independent updates.” The two difficulties are retargeted separately, each from the production rate of its own block type. η (eta) is only the initial ratio d_r/d_b; it is not enforced afterward.

Base difficulty d_b

After every CLASSICAL_WINDOW classical blocks, d_b retargets to hold classical production at CLASSICAL_BLOCK_TIME per block:

t* = (elapsed over the window) / CLASSICAL_WINDOW
d_b ← clamp( d_b · CLASSICAL_BLOCK_TIME / t*,  d_b/x,  d_b·x ),  floored at MIN_DIFFICULTY

Reduced difficulty d_r

After every SOLUTION_WINDOW solution blocks, d_r retargets to hold solution production at SOLUTION_BLOCK_TIME per block (same formula, solution timing). If solutions arrive too fast — e.g. on an easy instance — this drives d_r back up.

Drought rule

If CLASSICAL_WINDOW consecutive classical blocks pass with no solution, d_r is cut by the max factor x (floored at MIN_REDUCED_DIFFICULTY). So as a problem hardens, the discount deepens — and hoarding a solution risks someone else publishing first at an ever-cheaper d_r. Every update is clamped to a factor of x per step.

8Epochs & problem rotation

The genesis block defines a corpus — a problem type, a seed, and a size — rather than a single puzzle. Instance i is generate("{seed}:i:{i}"). The chain optimizes one instance per epoch.

Rotation

An epoch ends when the instance saturates: SATURATION_WINDOW consecutive blocks find no improvement (a hard MAX_EPOCH cap is a backstop). The next instance index is derived from the hash of the rotating block:

index ← int( rotating_block.hash, 16 ) mod CORPUS_SIZE

This is deliberately unpredictable (you cannot pre-compute which instance comes next, so you cannot stockpile solutions) and unchooseable (miners cannot cherry-pick the easiest fresh instance). Holding an instance for a whole epoch — rather than switching every block — is what lets it be optimized to depth.

Why d_b is global but d_r resets

d_b tracks network hashrate and security, so it is continuous across epochs. d_r is the per-problem discount and resets to d_b·η on every rotation. The reset matters: a fresh instance is trivially improvable, so without it a miner could farm cheap discounted blocks on each new instance. Resetting d_r forces the discount to re-earn itself (via droughts) as the new instance hardens.

Per-instance memory

The chain remembers the best score reached for each instance. Revisiting an instance does not reset it — a new solution must still beat that instance’s recorded history.

9Transactions & accounts

crick uses an account model (balances + per-sender nonces), not UTXOs.

  • A transaction is {sender, recipient, amount, nonce, pubkey, signature}. Its id is sha256d(payload + signature), where payload is the canonical JSON of everything but the signature.
  • The sender’s nonce must equal the number of transactions they have already sent (replay protection), the amount must be positive, and the sender must have sufficient balance.
  • The first transaction in every block is the coinbase: sender "coinbase", amount BLOCK_REWARD, and its nonce equals the block height (keeping coinbase ids unique). No other coinbase is allowed.

Balances and nonces are derived state: replaying blocks from genesis reconstructs them exactly.

10Consensus & fork choice

The valid chain is the one with the most accumulated work, defined as the sum of every block’s declared difficulty:

totalWork = Σ block.difficulty

A node adopts a peer’s chain only if it shares the same genesis hash and has strictly greater total work — and only after fully re-validating it from genesis (every signature, difficulty, solution, epoch transition, and balance). Note that solution blocks contribute their reduced difficulty to total work, so cheap solution blocks add less security weight than classical ones.

11Networking (P2P)

Nodes gossip over plain HTTP/JSON. Peers are seeded manually with --peer.

EndpointPurpose
GET /statusHeight, tip, d_b, d_r, best score, peers, balance.
GET /chainThe full chain as JSON (used for sync).
GET /solutionThe best solution recorded for the active instance.
POST /blocksSubmit a new block; on a fork/gap the node syncs from the sender.
POST /txSubmit a signed transaction to the mempool.
GET/POST /peersList / announce peers.

When a node mines or accepts a new tip it broadcasts the block to its peers. If an announced block does not extend the tip, the node fetches the sender’s chain and applies the fork-choice rule above.

12Wallets & signatures

Keys are ECDSA on secp256k1 (the Bitcoin curve), signed deterministically. An address is derived from the public key:

address = "crk1" + sha256( pubkey_bytes ).hex()[:40]

A transaction is valid only if its pubkey hashes to its sender address and the signature verifies against the canonical payload. Wallets are stored locally; the private key never leaves the node.

13Security model

  • 51% attacks apply as in any PoW chain: an entity with majority work can reorg. Solution blocks carry reduced weight, so a network where most blocks are solution blocks has proportionally less hashing security — a tension the operator tunes via η and the windows.
  • Bubka attack (hoarding a run of successive solutions to win many blocks or fork cheaply): mitigated by keeping SOLUTION_WINDOW small, so rapid solution blocks quickly retarget d_r upward, plus the unpredictable next-instance selection (you can’t stockpile solutions for an instance you can’t predict).
  • Cheap-instance farming: prevented by resetting d_r each epoch and by making the active instance unchooseable (hash-selected), so a miner cannot keep hopping to trivially-improvable instances at a deep discount.
  • Solution copying / front-running: a copied solution never beats the recorded best, so it is invalid; the strictly-better rule makes plagiarism worthless on its own.
  • Determinism is consensus-critical: verification and scoring are integer-only so all nodes agree exactly. Any floating-point or nondeterministic verifier would fork the chain.

14Block validation algorithm

A block extending the tip is accepted only if every check passes, in order:

# structural
1.  height == tip.height + 1
2.  prev_hash == tip.hash
3.  problem field is null            # only genesis carries the corpus spec
4.  timestamp ≥ tip.timestamp − MAX_BACKWARD_DRIFT
5.  timestamp ≤ now + MAX_FUTURE_DRIFT

# which difficulty must this block meet?
6.  if block has a solution:
        verify(solution) is true
        score(solution) > best score for the active instance
        required = d_r
    else:
        required = d_b
7.  block.difficulty == required
8.  int(block.hash, 16) < target(required)

# transactions
9.  tx[0] is the coinbase (amount == BLOCK_REWARD, nonce == height); no other coinbase
10. each other tx: valid signature, amount > 0, nonce == sender's count, sufficient balance

# on success: append, update difficulty schedule, maybe rotate the epoch

16Determinism & limitations

This is a research prototype. Known limitations:

  • Synthetic instances. For now docking and mcs-protein instances are synthesized deterministically from the seed so the package runs with no external data. Real deployment loads ESM Atlas contact maps / Enamine ligands into the same instance spec — the verifier and scorer are unchanged.
  • Instance generation PRNG. Instance synthesis uses Python’s RNG, so cross-language nodes would need a specified PRNG. The consensus path (verify/score) is already pure integer arithmetic and is language-agnostic.
  • Networking. HTTP gossip with manual peer seeding — no peer discovery, NAT traversal, or encrypted transport.
  • Single chain head. Fork choice re-validates whole chains rather than maintaining a header tree; fine at research scale.

See the source for the authoritative behavior, and the two papers for the underlying theory.
